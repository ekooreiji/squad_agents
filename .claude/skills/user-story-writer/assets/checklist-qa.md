# ✅ Checklist de Qualidade — User Story

Use este checklist para validar se sua user story está pronta para desenvolvimento.

---

## 📋 Estrutura & Completude

### User Story Base
- [ ] **Tem titulo descritivo**: US-XXX: [Descrição clara]
- [ ] **Persona identificada**: "Como [role/persona]..."
- [ ] **Ação clara**: "Quero [verbo + ação]..."
- [ ] **Benefício documentado**: "Para [valor/razão]..."

### Status & Informações
- [ ] **Status definido**: Draft, Ready, In Progress, ou Done
- [ ] **Prioridade clara**: Must Have, Should Have, Nice to Have

---

## ✅ Critérios de Aceitação — Formato BDD

### Completude
- [ ] **Mínimo 2 cenários**: Fluxo principal + 1 variação
- [ ] **Cada cenário tem nome descritivo**: "Cenário 1: ..." ✅

### Estrutura Given/When/Then
- [ ] **"Dado" (Given)**: Contexto/precondição clara
- [ ] **"Quando" (When)**: Ação específica do usuário
- [ ] **"Então" (Then)**: Resultado esperado mensurável

---

## ☑️ Critérios de Aceitação — Formato Checklist

### Completude
- [ ] **Mínimo 5 critérios**: Base da funcionalidade
- [ ] **Máximo 15 critérios**: Não virar lista de tarefas
- [ ] **Inclui critérios de erro**: "Se X, então Y"

### Qualidade
- [ ] **Objetivo e específico**: Não é uma opinião
- [ ] **Testável**: QA consegue verificar
- [ ] **Mensurável**: Resultado claro (✅/❌)

---

## 📌 Regras de Negócio & Contexto

- [ ] **Regras documentadas**: Restrições, limites
- [ ] **Dependências mencionadas**: "Requer US-XXX"
- [ ] **Casos excepcionais listados**: O que fazer se falhar
- [ ] **Nenhuma ambiguidade**: Ou tudo está clarificado

---

## 🎯 Dev & QA podem começar?

### Do Ponto de Vista do Dev
- [ ] **Consigo começar?** Falta informação?
- [ ] **Tenho critérios de parada claros?**
- [ ] **Há ambiguidades que impedem?**

### Do Ponto de Vista do QA
- [ ] **Consigo testar cada critério?**
- [ ] **Resultado esperado está claro?**

---

- [ ] ✅ Passou em **Critérios de Aceitação**
- [ ] ✅ **Regras & Contexto** claros
- [ ] ✅ **Dev consegue implementar**
- [ ] ✅ **QA consegue testar**
