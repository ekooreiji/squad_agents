# Template Validation Checklist

Use esta checklist para validar cada documento antes de finalizar.

---

## Épico (Epic)

### Completude
- [ ] ID único atribuído
- [ ] Título descritivo
- [ ] Problema/oportunidade claramente descrito
- [ ] Escopo definido (inside/outside)
- [ ] Hipóteses listadas
- [ ] User stories vinculadas

### User Stories
- [ ] Todas as stories com criteria de aceite
- [ ] Priorização MoSCoW aplicada
- [ ] Estimativas presentes
- [ ] Dependências identificadas

### Execução
- [ ] Critérios de aceite do épico definidos
- [ ] Métricas de sucesso estabelecidas
- [ ] Riscos identificados
- [ ] Timeline com marcos

### Revisão
- [ ] Revisado pelo product owner
- [ ] Revisado pelo tech lead
- [ ] Aprovação documentada

---

## Release Notes

### Formato
- [ ] Versão correta
- [ ] Data de release准确
- [ ] Tipo de release identificado

### Conteúdo
- [ ] Highlights presente (max 2 linhas)
- [ ] Novos recursos descritos
- [ ] Melhorias listadas
- [ ] Bug fixes com IDs
- [ ] Mudanças de comportamento destacadas
- [ ] Deprecações documentadas

### Qualidade
- [ ] Linguagemuser-friendly
- [ ] Termos técnicos explicados quando necessário
- [ ] Links funcionando
- [ ] Known issues listados

### Revisão
- [ ] Revisado por produto
- [ ] Revisado por suporte
- [ ] Aprovação de comunicação

---

## FAQ

### Completude
- [ ] Categorias organizadas
- [ ] Perguntas relevantes para o público-alvo
- [ ] Respostas precisas e atualizadas

### Qualidade
- [ ] Linguagem clara e acessível
- [ ] Consistência de formato
- [ ] Sem duplicatas
- [ ] Links verificados

### Estrutura
- [ ] Categoria: [P] pergunta / [R] resposta
- [ ] Contato de suporte incluso
- [ ] Data de atualização

---

## Roadmap

### Estratégia
- [ ] Visão geral claro
- [ ] Princípios estratégicos definidos
- [ ] Alinhamento com business goals

### Planeamento
- [ ] Trimestres claramente definidos
- [ ] Features priorizadas
- [ ] Datas realistas
- [ ] Status atualizado

### Conteúdo
- [ ] Temas principais por trimestre
- [ ] Features categorizadas por tipo
- [ ] Métricas establecidas
- [ ] Items long-term considerados

### Transparência
- [ ] Mudanças do último roadmap documentadas
- [ ] Feedback channels jelas
- [ ] Notas adicionais se necessário

---

## User Story

### Estrutura
- [ ] ID único
- [ ] Formato "Como... Eu quero... Para que..."
- [ ] Critérios de aceite claros

### Cobertura
- [ ] Happy path coberto
- [ ] Casos de borda considerados
- [ ] Casos de erro definidos

### Definition of Done
- [ ] Todos os itens revisados
- [ ] Tasks definidas

---

## Feature Spec

### Requisitos
- [ ] Requisitos funcionais claros
- [ ] Requisitos não-funcionais com thresholds
- [ ] Casos de borda/erro documentados

### Diseño
- [ ] Arquitetura propuesta
- [ ] Fluxo de dados descrito
- [ ] API endpoints definidos

### Testing
- [ ] Testes unitários planejados
- [ ] Testes de integração considerados
- [ ] Testes E2E identificados

### Revisão
- [ ] Revisado por engenharia
- [ ] Revisado por QA

## Changelog

### Formato
- [ ] Versão e data corretas
- [ ] Tipos遵循 convenção (Added, Changed, etc.)
- [ ] Ordem cronológica (mais recente primeiro)

### Conteúdo
- [ ] Todas as mudanças inclusas
- [ ] Links para issues/PRs
- [ ] Deprecações documentadas

### Qualidade
- [ ] Sem duplicatas
- [ ] Descrições concisas
- [ ] Links funcionando
